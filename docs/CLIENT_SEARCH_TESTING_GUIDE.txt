This document explains how you can test the appointment search feature, what
you should see, and what is considered correct behaviour. You can share this
file with your team or testers as-is.


--------------------------------------------------------------------------------
1. WHAT IS LIVE NOW
--------------------------------------------------------------------------------

• The public search (home page / user dashboard) uses real eZdrav waitlist
  data stored in our database (not CSV files anymore).

• The user types what they need in plain language (Slovenian or English),
  picks a city, and the system:
  1) Uses AI to understand the request and pick relevant medical keywords
  2) Finds matching clinics/hospitals in that city
  3) Shows providers grouped by institution, with services and waiting info

Notes for testers:

• English and Slovenian both work (e.g. “dermatology skin” → dermatology services).

• Empty result is OK when that service does not exist in the chosen city in eZdrav.

• Tests 19–20 are important: the system should NOT return random hospitals for
  gibberish or a single word “pregled” alone. User should see empty results and
  an explanation asking them to describe their need more specifically.

• We validated 20 automated scenarios: 20 PASS, 0 warnings (May 2026).


--------------------------------------------------------------------------------
5. HOW TO TEST — ADMIN “HOSPITAL SCRAPE” PAGE
--------------------------------------------------------------------------------

For your team reviewing raw data:

1. Log in as admin.

2. Open: Admin → Hospital scrape (eZdrav).

3. Use filters: service, urgency, region, city, search box.

4. Table shows: appointment/waiting summary, city, hospital, address, phone, etc.

5. Click any row/cell → modal with full details.

6. Pagination and sorting should work on large datasets.

This confirms the database import; it does not replace testing the public AI search.


--------------------------------------------------------------------------------
6. WHAT GOOD RESULTS LOOK LIKE (USER PERSPECTIVE)
--------------------------------------------------------------------------------

Good:

• Query “MRI” + Ljubljana → several providers; service names mention MRI.

• Query “psihiatrija” + Maribor → multiple psychiatric services listed.

• Query “fizioterapija” + Celje → physiotherapy-related service names.

• Nonsense query → zero hospitals, message that we could not understand.

Not a bug (usually):

• Zero results in a city where eZdrav has no listing for that specialty.

• Fewer providers than on the national eZdrav site (we only show imported,
  non-unavailable rows; max 50 service rows per search for performance).

• Same provider appearing once with several services (grouped by institution).

7. TECHNICAL TESTING (FOR DEVELOPERS / STAGING)
--------------------------------------------------------------------------------

API endpoint:

  POST /api/search
  Content-Type: application/json
  Body: { "query": "MRI", "city": "Ljubljana" }

Example (replace host if needed):

  curl -X POST http://localhost:5000/api/search \
    -H "Content-Type: application/json" \
    -d '{"query":"MRI","city":"Ljubljana"}'

Response should include:

  intent, explanation, hospitals[], totalHospitals, filterCity

Automated regression (on server machine, from server folder):

  node scripts/test-search-validation.js

Expect summary line: 20 PASS, 0 WARN, 0 FAIL

Cities list:

  GET /api/search/locations


--------------------------------------------------------------------------------
8. KNOWN LIMITATIONS (SET EXPECTATIONS)
--------------------------------------------------------------------------------

• Search depends on OpenAI — brief outages or missing API key block search.

• Data is a snapshot from eZdrav CSV import; not live-synced every minute.

• Very generic words alone (“pregled”, “doctor”) are intentionally rejected
  to avoid showing the entire city catalogue.

• Some English terms map to Slovenian service names; rare terms may return empty.

• Appointment dates in results come from scraped “appointment summary” text,
  not a live booking API from hospitals.
